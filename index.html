Enfim, o que é o Git, e para que serve?

"O Git é uma ferramenta para controle de versão de arquivos, utilizado para controlar versoes de códigos-fonte e desenvolvimento colaborativo." Fonte: GeekHunter

Criado em 2005 por Linus Torvalds, o Git facilita através do uso de uma ferramenta de CLI (Command Line Interface), ou seja, uma Interface de Linguagem de Comandos, o versionamento de códigos. O versionamento permite que os programadores saibam quando e quais alterações foram feitas no código.

Normalmente, o Prompt de comando pode ser utilizado para iniciar o processo de versionamento, porém o próprio Git fornece duas interfaces para serem utilizadas, o Git Bash, que é uma Command Line Interface, e o Git GUI que se trata de uma Interface gráfica.

A principal diferença entre o Git e o Github se encontra em como esses arquivos serão armazenados. No Git os repositórios são armazenados de maneira local, ou seja na maquina do desenvolvedor, enquanto o Github conta com um metodo de armazenamento em Nuvem, também pode se dizer que ele é "Um sistema distribuído é uma coleção de programas de computador que utilizam recursos computacionais em vários pontos centrais de computação diferentes para atingir um objetivo comum e compartilhado. Os sistemas distribuídos visam remover gargalos ou pontos centrais de falha de um sistema. Fonte: Atlassian

Git por baixo dos panos

Nesta etapa entraremos em um conceito mais abstrato e muito dificil de se resumir, portanto tentarei ser o mais breve possivel.

Antes de entrarmos nos conceitos de objetos, precisamos primeiro esclarecer como o Git percebe uma modificação em seus arquivos.

SHA1

O SHA1 básicamente é um algoritimo de encriptação, baseado em uma string de 40 caracteres aleatórios. O Git utiliza este algoritmo para associa-lo à um arquivo ou diretorio. Sempre que algum arquivo ou diretório é modificado, sua referência, ou seja, seu hash também muda.

A forma como o git funciona por baixo dos panos é através de objetos, e sobre como estes afetam os demais em sua cronologia. Basicamente se uma alteração for feita no objeto de nivel mais baixo, todos os demais objetos "parentes" deste que recebeu a modificação sofrerão uma alteração tambem.

Os objetos fundamentais são divididos como:

Blob: básicamente, de forma rudimentar, o objeto atuaria como um arquivo para algumas informações que se alguma delas sofre alteração, consequentemente seus parentes também serão alterados.

Tree: funciona como um "container" para os demais objetos, podendo ser eles blobs, ou outras trees. Assim como o bloob, se algum metadado de uma tree for alterado, a referência, ou seja, o SHA1 da tree também sera modificado.

Primeiros comandos com o git

Agora que você ja sabe o básico sobre como o git funciona por baixo dos panos (realmente o básico do básico, sugiro que veja as aulas do bootcamp), vamos dar inicio ao primeiro repositório local. Nesta sessão tambem serão apresentados os comandos para empurrar seus codigos para o GitHub.

Git Bash
git init Inicia um repositório local
git Status Exibe o estado atual do seu repositório, pendentes ou não de staging
git add Comando para adicionar arquivos para a área de staging
git branch -M "branch_name" Renomeia a branch atual
git commit -m "nome_commit" Envia os arquivos do stage para o status de tracked
git remote add origin "link do repositório github Cria o vínculo entre o armazenamento local e o github
git push -u origin branch_name Empurra os arquivos commitados para o github
git pull Transfere os arquivos armazenados no Github para a maquina local

Introdução ao GitHub

Neste tópico aprenderemos a criar o repositório no github para receber os arquivos locais da maquina. Por ser uma interface visual, despensa muitas explicações (tentarei ser breve).

Na página inicial do GitHub (Assumindo que o leitor ja tem uma conta criada), no canto superior esquerdo, abaixo da barra de pesquisas do GitHub, clique em New.

A partir dessa parte ja é muito intuitívo, por se tratar de uma interface gráfica meio que despensa muita explicação. Preencha os dados e clique em Create Repository.

Pronto! Seu repositório esta criado, agora é só pegar sua URL, utilizar o comando para criar a conexão entre o GitHub e a maquina local.

Resolvendo Conflitos

Nesta sessão veremos como o Git responde aos "conflitos".

O Github permite que vários desenvolvedores trabalhem no mesmo código simultaneamente, e isso é uma vantagem enorme quando se trata do tempo em que uma aplicação sera desenvolvida, porém existe um caso que é muito conhecido e também é algo muito comum de se acontecer quando se pensa em algum projeto muito grande e com muitos desenvolvedores trabalhando em uma mesma linha de código, o que gera o conflito.

Quando o conflito ocorre, é necessario o desenvolvedor fazer o Pull dos arquivos no repositório do GitHub.

Neste caso o git não tomará nenhuma atitude pelo desenvolvedor, ele simplesmente devolve os casos que geraram o conflito para que o especialista tome a decisão sobre o que fazer com o conflito.
